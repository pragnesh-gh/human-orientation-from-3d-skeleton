1. PIPELINE ARCHITECTURE INSIGHTS (not detailed in report)
================================================================================

1.1 Timeline Preservation (gen_kpts.py)
    The 2D detection stage (YOLO+SORT+HRNet) inserts ZERO-KEYPOINT placeholder
    frames whenever no person is detected, rather than skipping frames.
    WHY: PoseFormerV2 is a temporal model (243-frame window). Dropping frames
    would compress the timeline and create false temporal relationships.
    The downstream code uses a valid_mask to distinguish real vs placeholder frames.
    This is a subtle but critical design choice — without it, the model would
    "see" two distant moments as adjacent, producing hallucinated motion.

1.2 Detection Fallback (gen_kpts.py)
    When YOLO fails on the current frame, the pipeline reuses the PREVIOUS
    frame's bounding box and detection score. This prevents single-frame
    detection dropouts from breaking the tracking chain.
    TRADE-OFF: Adds one frame of detection latency in edge cases, but prevents
    the far worse problem of losing person identity (SORT ID reassignment).

1.3 Flip Augmentation at Inference (vis.py)
    Every frame is processed TWICE by PoseFormerV2 — once normally, once with
    horizontal flip (x-coordinates negated, left/right joints swapped).
    The two predictions are averaged after un-flipping.
    WHY: This is test-time augmentation that reduces left-right ambiguity,
    a known weakness of monocular 3D pose estimation (the model can confuse
    which leg is in front). Costs 2x compute but measurably improves accuracy.

1.4 Edge Padding Strategy (vis.py)
    At video boundaries (first/last ~121 frames), the 243-frame temporal
    window cannot be fully filled. The code uses numpy "edge" padding —
    replicating the boundary frame to fill missing slots.
    ALTERNATIVE considered: zero-padding (worse — model sees frozen-to-moving
    transition). Mirror padding (complex, no clear benefit).
    CONSEQUENCE: First and last ~5 seconds of output may be slightly less
    accurate due to reduced temporal context.

2. ORIENTATION ESTIMATION INSIGHTS (supplements report Section III-F)
================================================================================

2.1 Gram-Schmidt Asymmetry
    The orthogonalization is NOT symmetric (unlike SVD-based approaches).
    Priority order: shoulder axis (RIGHT) is pinned exactly, then FORWARD
    is derived via cross product, then UP is recomputed to enforce orthogonality.
    WHY: Shoulder width is the most consistently observable body measurement
    across viewpoints and occlusion levels. Anchoring to it produces more
    stable orientation estimates than treating all axes equally.
    CONTRAST: SVD/Kabsch would find the "best-fit" orthogonal frame, but
    no single axis would match exactly. For robotics (where LEFT vs RIGHT
    matters), the Gram-Schmidt approach is more predictable.

2.2 Dual Confidence Model
    The confidence score blends TWO independent signals (50/50 weight):
    - Joint detection confidence (from HRNet): "how sure is the detector?"
    - Geometric frame stability: "how well-formed is the torso frame?"
    The geometric term penalizes:
    - Non-orthogonality: (right·up)² + (up·forward)² + (forward·right)²
    - Vector collapse: when normalized vectors shrink toward zero
    WHY: A detector can be confident about joint positions that still produce
    a degenerate torso frame (e.g., person seen edge-on, shoulders nearly
    collinear with spine). The geometric term catches this.

2.3 SLERP-EMA vs Naive Smoothing
    Temporal smoothing uses Spherical Linear Interpolation (SLERP), not
    simple quaternion averaging.
    WHY: Naive averaging (q_smooth = 0.8*prev + 0.2*cur) does NOT preserve
    unit norm. The result would need re-normalization and can produce
    incorrect rotations near antipodal quaternions.
    SLERP traverses the shortest arc on the 4D unit sphere by construction.
    DETAIL: When quaternions are nearly parallel (dot > 0.9995, ~1.8 degrees),
    the code falls back to LERP+normalize to avoid division-by-near-zero
    in the sin(theta) denominator.

2.4 Confidence-Gated Hold
    When orientation confidence drops below a threshold (default 0.5),
    the system HOLDS the previous quaternion instead of updating.
    WHY: During heavy occlusion, the computed orientation may flip wildly
    between frames. Holding the last confident estimate is more useful for
    downstream robotics than a noisy update. The held value is still
    reported with reduced confidence in the log.

3. 2D KEYPOINT PROCESSING INSIGHTS (not in report)
================================================================================

3.1 COCO-to-H36M Joint Synthesis
    The H36M skeleton has joints that don't exist in COCO-17 (pelvis, spine,
    thorax, head). These are SYNTHESIZED by averaging:
    - Pelvis = mean(left_hip, right_hip)
    - Spine = mean(all four torso joints)
    - Thorax = mean(shoulders) adjusted 1/3 toward neck
    - Head = mean(COCO joints 1-4) with y-adjustment
    PLUS three empirical coordinate tweaks:
    - Head y pulled toward shoulder midpoint (1/4 factor)
    - Chest x magnified from head-pelvis axis (2x extrapolation)
    - Thorax y adjusted by neck-to-head distance (2/3 factor)
    These are HAND-TUNED corrections, not learned. They compensate for
    systematic differences between COCO and H36M joint definitions.

3.2 Kinetic Chain Repair (revise_kpts)
    When joint confidence < 0.3, the pipeline repairs missing joints using
    anatomical parent-child relationships:
    - Missing knee → copied from hip position
    - Missing ankle → copied from knee position
    - Both sides missing → symmetric copy from the other side
    WHY: PoseFormerV2 handles noisy inputs better than MISSING inputs.
    A knee placed at the hip is wrong but plausible; a zero-coordinate
    knee creates impossible geometry that confuses the temporal model.

3.3 Quality Gate (NEW — vis.py, added during this session)
    A runtime check that rejects 2D keypoints where:
    - >50% of joints are outside the image bounds (with 50px margin)
    - All joints are zero (placeholder frame)
    - Maximum joint spread < 15px (skeleton collapsed to a point)
    Rejected frames are treated as no-person (plain RGB, held 3D pose).
    WHY: HRNet can output "confident" keypoints even when the person is
    barely visible (1-2 body parts at frame edge). These produce wild
    2D overlays and distorted 3D skeletons. The quality gate catches
    these before they reach the model.

4. POSEFORMERV2 MODEL INSIGHTS (supplements report Section III-E)
================================================================================

4.1 Dual-Stream Architecture
    The model processes input through TWO parallel streams:
    - Spatial stream: attention across joints within each frame
      ("which joints relate to which?")
    - Spatio-temporal stream: DCT on the temporal axis, then attention
      ("how does motion evolve over time?")
    These streams are processed by MixedBlocks that maintain separation
    internally — frequency tokens get FreqMlp, spatial tokens get standard MLP.

4.2 DCT as Motion Representation
    The Discrete Cosine Transform converts the temporal joint trajectory
    into frequency coefficients. Only the first K coefficients are kept
    (default K=27 out of 243).
    WHY: Human motion is naturally LOW-FREQUENCY — walking, turning, sitting
    are smooth. High-frequency components are mostly noise/jitter from the
    2D detector. Truncating to K=27 is an implicit temporal smoothing
    that's LEARNED rather than hand-designed.
    ANALOGY: Like JPEG compression for motion — keep the important
    structure, discard the noise.

4.3 Learned Weighted Pooling
    The output head uses Conv1d(kernel_size=1) instead of mean pooling.
    This learns WHICH temporal frames/frequency coefficients matter most
    for the final prediction, rather than treating all equally.

5. DEPTH & METRIC TRANSLATION INSIGHTS (supplements report Section III-G)
================================================================================

5.1 Depth Only Affects Logging, Not Visuals
    Adding --translation-source depth computes pos_xyz_m in the log file
    but does NOT change any visual output (2D overlay, 3D skeleton, video).
    The depth-based position and the skeleton are independent:
    - Skeleton: root-relative, from PoseFormerV2 (unitless)
    - Position: absolute, from depth back-projection (meters)
    They're combined only in the log for downstream ROS consumption.

5.2 Robust Depth Sampling
    Depth is sampled using a 7x7 MEDIAN window at the pelvis pixel,
    ignoring zero values (invalid depth readings).
    WHY: Depth sensors produce noisy measurements, especially at edges
    and on dark/reflective surfaces. Median filtering in a window is
    more robust than single-pixel sampling. Zeros (sensor failures)
    would corrupt a mean but are harmlessly excluded from the median.

5.3 Frame Alignment Convention
    All positions are in the camera optical frame (x=right, y=down, z=forward).
    This differs from the common robotics convention (z=up) but matches
    the sensor's native frame. The choice avoids an extra rotation and
    makes visual debugging easier ("z = distance from camera").

6. ROS INTEGRATION INSIGHTS (not detailed in report)
================================================================================

6.1 Last-Known-Value (LKV) State Machine
    The ROS publisher implements a 5-state machine:
    - ok: fresh pose, publish immediately
    - stale: no current detection, republish last-known if age < timeout
    - no_person: no detection AND no last-known value
    - no_depth: person detected but no depth reading
    - timeout: last-known value too old, stop publishing
    WHY: Robotics systems need CONTINUOUS pose updates for planning.
    A 100ms gap in detection shouldn't make the robot "forget" where the
    human is. The stale republish bridges brief detection gaps.

6.2 Diagnostics Topic
    A separate /human_pose_status topic publishes JSON diagnostics on
    EVERY frame (not just successful ones):
    { status, confidence, translation_source, note, age_ms, frame_id }
    WHY: Downstream nodes can distinguish "person moved" from "sensor
    failed" from "person left" — enabling appropriate robot behavior
    for each case (wait, search, abort).

6.3 Replay Tool (replay_pose.py)
    The log replayer parses pose_stamped.log and republishes to ROS topics
    at the original frame rate. It implements the same LKV/stale logic,
    so replayed data behaves identically to live data.
    USE CASE: Teammates on ROS/Linux can replay logged data in RViz
    without needing the full pipeline or GPU.

6.4 ROS Bag Extraction (export_rgbd_from_bag.py)
    Color and depth frames from a ROS bag are paired using TOLERANCE-BASED
    timestamp matching (default 30ms window) rather than exact matching.
    WHY: RealSense color and depth streams run on independent hardware
    clocks. Exact timestamp matching would miss most frames. The 30ms
    tolerance handles typical sensor clock skew.
    FEATURE: Auto-continuation — if extraction is interrupted, the tool
    detects the highest existing frame index and resumes from there.

7. SYSTEM-LEVEL INSIGHTS FOR PRESENTATION
================================================================================

7.1 Windows-First Development
    The entire pipeline runs on Windows (no ROS required). ROS features
    (publishing, TF, markers) are behind try/except guards and gracefully
    degrade to console output + log files. This enabled rapid iteration
    without a Linux/ROS environment.

7.2 Robustness Through Graceful Degradation
    Every stage has a fallback:
    - YOLO fails → reuse previous detection
    - HRNet fails → insert zero-keypoint placeholder
    - Low confidence → hold previous orientation
    - No depth → log position as NA
    - No ROS → write to file instead
    - Quality gate fails → treat as no-person frame
    The system NEVER crashes on bad input; it always produces output
    (possibly degraded) and logs what went wrong.

7.3 Separation of Concerns
    Three independent outputs that serve different consumers:
    - Visual (pose2D/, pose3D/, video) → human inspection
    - Log (pose_stamped.log) → offline analysis, replay
    - ROS topics (/human_pose, /human_pose_status) → live robotics
    Each can be enabled independently. The core pipeline doesn't know
    which outputs are active.

7.4 Key Numbers for Presentation
    - Temporal window: 243 frames (~10 seconds at 24fps)
    - Frequency coefficients kept: 27 out of 243 (89% compression)
    - Joint count: 17 (H36M format, converted from COCO-17)
    - Orientation axes: 3 (forward/right/up from pelvis)
    - Depth sampling: 7x7 median window
    - Smoothing: SLERP-EMA with alpha=0.6
    - Quality gate: >50% OOB joints OR <15px spread → reject
    - Camera: RealSense, 1280x720, fx=911.47, fy=911.56
    - Stale timeout: 1000ms for ROS republishing

8. POTENTIAL PRESENTATION TALKING POINTS
================================================================================

8.1 "Why pelvis, not head?"
    The report covers this well (Section III-F), but a strong slide would
    show: head turns independently of body, pelvis+spine is the torso's
    structural axis. For robot approach planning, BODY facing > HEAD facing.
    Head can look at phone while body faces door.

8.2 "Why frequency domain?"
    Human motion is smooth — walking is ~2Hz, turning ~1Hz. The DCT
    naturally captures this. Keeping 27 of 243 coefficients is like keeping
    the melody and discarding the noise. This is the key insight of
    PoseFormerV2 over PoseFormer (which used raw temporal attention).

8.3 "What breaks the system?"
    Three failure modes, clearly ordered:
    1. No skeleton at all (HRNet can't find person) → SKIP
    2. Skeleton present but garbage (partial visibility) → Quality gate
    3. Skeleton plausible but wrong (left-right flip) → Flip augmentation
    Show one example of each from your results.

8.4 "Why not just use a depth camera for everything?"
    Depth gives POSITION (where is the person) but not ORIENTATION
    (which way are they facing). You need the 2D→3D skeleton pipeline
    to infer body structure, then the torso frame computation for facing.
    Depth provides the metric anchor; the skeleton provides the semantics.
