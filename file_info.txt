0) End-to-End Methodology Overview (what your system does)
======================================================================

Your project’s practical pipeline (the demo/ workflow) is:

(ROS bag)
  └─ demo/export_rgbd_from_bag.py
       → color PNG sequence + depth PNG sequence (aligned by nearest timestamp)

(color frames)
  └─ demo/lib/hrnet/gen_kpts.py
       YOLOv3 person detection → SORT tracking → HRNet keypoints
       → 2D joints (COCO-17) + per-joint confidence over time

(2D joints)
  └─ demo/lib/preprocess.py
       COCO-17 → H36M-17 joint mapping + synthetic torso joints + low-conf joint repair
       → stable 2D tracks in the joint layout PoseFormer expects

(2D tracks, normalized)
  └─ demo/vis.py + common/camera.py
       normalize_screen_coordinates() + test-time flip augmentation
       → PoseFormerV2 inference (common/model_poseformer.py)
       → 3D joints (camera frame)

(3D joints)
  └─ common/orientation.py + common/quaternion.py
       torso coordinate frame → quaternion (wxyz) + confidence
       optional SLERP smoothing (temporal)

(Optional) metric translation (depth)
  └─ demo/vis.py
       pelvis pixel (u,v) + depth(u,v) + intrinsics (fx,fy,cx,cy)
       → backproject to (X,Y,Z) meters in camera frame

Outputs (for report + ROS)
  ├─ Visualization frames/videos (pose2D/ pose3D/ composite)
  ├─ input_2D/keypoints.npz  (reconstruction + valid_mask)
  ├─ pose_stamped.log        (per-frame pose + status + translation source)
  └─ demo/replay_pose.py republishes pose_stamped.log into ROS topics


======================================================================
1) demo/export_rgbd_from_bag.py  — RGB-D extraction from ROS bag
======================================================================

Purpose:
- Convert a rosbag into a dataset-like folder structure of *paired* RGB and depth images.

Core idea:
- Color and depth messages have different timestamps.
- Maintain buffers Qc, Qd and match by nearest timestamp within tolerance.

Key mechanics:
- Read rosbag messages for color_topic + depth_topic within [start, end].
- On each new message:
    try match nearest timestamp in the other buffer within ±time_tol
    if match -> save both as the same index
- If no match -> push into that topic’s queue
- Keeps queue bounded (prevents memory blow-up)
- Supports:
    --start-secs, --duration
    --time-tol
    --stride (subsample matched pairs)
    --max-pairs
    auto-continue indexing (resume exports cleanly)

Outputs:
- out/color/color_000000.png, ...
- out/depth/depth_000000.png, ...
Notes:
- Depth is stored as 16UC1 PNG (typically raw mm units). Later, vis.py uses depth_scale.

2) demo/lib/yolov3/  — person detection (crop ROI for HRNet)
======================================================================

2.1) demo/lib/yolov3/darknet.py
Purpose:
- Build YOLOv3 model from Darknet .cfg and load Darknet .weights.

Key pieces:
- parse_cfg(): reads blocks from cfg into dict list
- create_modules(): compiles blocks into PyTorch nn.ModuleList
  Supports layer types:
    convolutional, upsample, maxpool, route (concat), shortcut (residual), yolo
- Darknet.forward():
    caches intermediate outputs for route/shortcut
    concatenates detections from multi-scale YOLO heads
- load_weights(): reads Darknet binary weights header + layer weights into PyTorch tensors

2.2) demo/lib/yolov3/human_detector.py
Purpose:
- Thin wrapper to run YOLOv3 and return “person” bounding boxes + confidences.

Main function:
- yolo_human_det(img, model=None, reso=416, confidence=...):
    preprocess image to YOLO input size
    forward pass → raw detections
    write_results() applies confidence threshold + NMS
    rescales boxes back to original image coordinates (undo letterbox)
    returns:
      bboxs[K,4] = [x1,y1,x2,y2]
      scores[K,1]

Role in methodology:
- Provides a stable person crop window so HRNet keypoint estimation is focused and robust.


======================================================================
3) demo/lib/hrnet/gen_kpts.py  — 2D keypoints over video (YOLO→SORT→HRNet)
======================================================================

Purpose:
- Generate 2D keypoints (COCO-17) and confidence scores per frame.
- Produces arrays shaped to match PoseFormer demo expectations.

Core pipeline per frame:
1) Read frame (OpenCV)
2) Detect person bbox with YOLOv3 (human_detector.py)
   - If YOLO misses: reuse previous bbox (bboxs_pre)
3) Track bbox with SORT
   - Provides temporal consistency (reduces jitter / ID swaps)
4) Crop/normalize person ROI
5) Run HRNet pose model on crop
6) Convert HRNet output to pixel-space joints for that frame

Output tensors:
- keypoints: (M, T, 17, 2)  [M = tracked persons, T = frames]
- scores:    (M, T, 17)

Important robustness patches (timeline integrity):
- If a frame cannot be read or no track exists:
    append zeros for that frame (do NOT skip time)
  This preserves 1:1 alignment with:
    - video frames
    - depth frames
    - valid_mask
    - pose logs

Methodology note:
- This “never skip frames” policy is crucial for synchronized RGB-D + pose pipelines.


======================================================================
4) demo/lib/preprocess.py  — COCO-17 → H36M-17 mapping + cleanup
======================================================================

Purpose:
- PoseFormer expects H36M-style 17 joints.
- HRNet outputs COCO-17.
- This file transforms, synthesizes missing torso joints, and repairs low-confidence legs.

Key functions:
- coco_h36m(keypoints):
    input:  (T, 17, 2) in COCO order (pixels)
    output: (T, 17, 2) in H36M order + valid_frames indices

    Steps:
    - Create “virtual” joints (head/thorax/pelvis/spine) using averages of face/shoulders/hips
    - Copy remaining joints into H36M order via index maps
    - Apply small empirical alignment tweaks (to match H36M style)
    - valid_frames = frames where sum(keypoints)!=0

- h36m_coco_format(keypoints, scores):
    input:  keypoints (N,T,17,2), scores (N,T,17)
    output: h36m_kpts (N,T,17,2), h36m_scores (N,T,17), valid_frames(list per person)

    Synthesizes confidence for virtual joints using means of contributing joints.

- revise_kpts(h36m_kpts, h36m_scores, valid_frames):
    Repairs unstable leg joints when score < 0.3
    Strategy: copy upstream joint positions to maintain plausible skeleton
    (prevents huge jumps that break temporal modeling)

Methodology role:
- This is the “format bridge” that makes 2D detections compatible with PoseFormerV2.


======================================================================
5) demo/vis.py  — main demo: 2D → 3D + visualization + logging
======================================================================

Purpose:
- Central script that:
  - runs/get 2D keypoints (via gen_kpts)
  - normalizes 2D (common/camera.py)
  - runs PoseFormerV2 3D lifting (common/model_poseformer.py)
  - renders output images/video
  - (optional) estimates torso orientation quaternion
  - (optional) estimates metric translation using depth PNGs + intrinsics
  - writes logs for ROS replay

Key internal stages (conceptual):

5.1) 2D stage (get_pose2D)
--------------------------
Output artifact:
- demo/output/<run>/input_2D/keypoints.npz

Format (important for chatbot + report):
- reconstruction: (1, T, 17, 2) float32
- valid_mask:     (T,) uint8
    valid_mask[t] = 1 → person present
    valid_mask[t] = 0 → no person (or missing frame)

Robust fallback behavior:
- If HRNet/YOLO pipeline fails or no person detected:
    writes all-zero reconstruction and all-zero valid_mask
    appends to skip.log explaining the skip
This guarantees downstream stages don’t crash and T stays consistent.

5.2) 3D stage (get_pose3D)
--------------------------
Inputs:
- input_2D/keypoints.npz
- raw RGB frames (video or images)

Preprocessing:
- input_2D_no = (T,17,2) pixel coords
- normalize_screen_coordinates(X,w,h):
    Xn = X/w*2 - [1, h/w]
- Builds a fixed-length temporal window around each frame:
    window = 243 frames (centered), with edge padding
- Test-Time Augmentation:
    - create flipped copy: x *= -1
    - swap left/right joints
    - run model on both
    - unflip 3D back, average outputs

Model inference:
- Loads PoseFormerV2 checkpoint (27_243_45.2.bin)
- output_3D: (1, 1, 17, 3) per frame (center frame prediction)

Coordinate conventions (as used in code comments):
- Camera optical-like frame:
    x = right
    y = down
    z = forward

World transform (for visualization only):
- camera_to_world(post_out, R=fixed_rot, t=0)
- then shifts z so min(z)=0 to keep skeleton above “ground”

Missing-person handling:
- If valid_mask[t]==0:
    keeps previous 3D pose (prev_post_out) to avoid sudden drops
    and logs status accordingly

5.3) Orientation estimation (optional)
--------------------------------------
Enabled by args.estimate_orientation.

Computed from the predicted 3D skeleton:
- Uses common/orientation.py:
    compute_torso_frame(joints_3d, layout={h36m|coco17})
    returns:
      quat_wxyz, forward, right, up, confidence

Smoothing:
- Confidence-gated SLERP-EMA:
    if confidence < conf_min and prev_quat exists:
        keep prev_quat (don’t update)
    else:
        prev_quat = slerp(prev_quat, cur_quat, alpha)

Outputs:
- orientation_rows buffer can be saved (CSV/JSON depending on flags)
- optional 3D triad overlay in rendered pose images

Important design note (current vis.py structure):
- The metric translation (depth) block is placed inside the orientation-enabled path.
  Practically: if you want depth translation, keep estimate_orientation enabled.

5.4) Metric translation from depth (optional)
---------------------------------------------
Enabled by:
  --translation-source depth
and depth inputs:
  --depth-path (image mode) OR --depth-dir + --depth-pattern (video/folder inference)

Computation:
- Choose pelvis/root joint pixel from overlay_2D_px (root_idx=0)
- Read aligned depth PNG (16UC1)
- Sample robust median depth in a small window around (u,v)
- Convert depth to meters:
    Z = depth_scale * Z_raw
- Backproject using intrinsics:
    X = (u - cx)/fx * Z
    Y = (v - cy)/fy * Z
    Z = Z
- pos_xyz_m is “camera-frame translation” in meters

Logs:
- translation_source: "depth" or "none"
- note explains missing depth, invalid depth sample, etc.

5.5) PoseStamped log (pose_stamped.log)
---------------------------------------
vis.py “stages” per-frame entries and writes one best line per frame (status-ranked).

Fields are designed to be replayable:
- frame_index
- has_person
- t_ms
- frame_id
- conf
- yaw_deg (optional)
- quat_wxyz (orientation)
- status (OK / SKIP:... with reason)
- pos_xyz_m (translation if available)
- translation_source + note

This log is the input to demo/replay_pose.py.

Methodology role:

- Lets teammates validate topics/TF in RViz without rerunning PoseFormer.


======================================================================
7) common/model_poseformer.py  — PoseFormerV2 (PoseTransformerV2)
======================================================================

Purpose:
- Transformer-based 2D→3D lifting model.

Inputs:
- x: (B, F, J, 2) normalized 2D joints
  typical: F=243, J=17

Key ideas:
1) Spatial modeling across joints (per selected frames)
2) Temporal modeling via combined:
    - time-domain tokens (kept frames)
    - frequency-domain tokens (DCT coefficients)
3) Fuse both in MixedBlocks, then aggregate → 3D pose

Main components:
- Joint_embedding: per-joint (2 → embed_dim_ratio)
- Spatial_pos_embed: learned (J tokens)
- Temporal DCT path:
    DCT across time → keep first Kc coeffs → embed → Temporal_pos_embed_
- MixedBlock:
    attention + split MLP:
      half sequence uses normal MLP
      half uses FreqMlp (DCT→MLP→iDCT)
- Weighted pooling:
    Conv1d over tokens to learn weighted mean for:
      frequency tokens and spatial tokens
- Head:
    Linear → (3J) → reshape to (B,1,J,3)

Output:
- 3D joints (relative), one pose per center frame

8) common/orientation.py + common/quaternion.py  — orientation + quaternion math
======================================================================

common/orientation.py
Purpose:
- Infer torso orientation from a 3D skeleton.

How:
- Use hips + shoulders to build torso axes:
    up      = normalize(spine - pelvis)
    right   = normalize(RShoulder - LShoulder)
    forward = normalize(cross(up, right))
    up      = normalize(cross(right, forward))
- Build rotation matrix [right | up | forward]
- Convert to quaternion (wxyz)

Returns:
- quat_wxyz
- basis vectors (forward/right/up)
- confidence score
- smooth_quat(prev,cur,alpha) uses SLERP

common/quaternion.py
Purpose:
- Torch-native quaternion ops used by camera/orientation:
    qrot, qinverse,
    quat_normalize, quat_dot, quat_slerp,
    quat_from_rotmat, rotmat_from_quat, basis_from_quat
Convention:
- quaternions are wxyz throughout this repo (be careful when writing to ROS messages)

13) Practical notes / gotchas (include these in methodology as “implementation details”)
======================================================================

- Timeline integrity:
  gen_kpts.py and vis.py intentionally avoid “dropping frames”.
  Missing detections are represented via valid_mask=0 and/or held-last-pose.

- Quaternion convention mismatch:
  Repo uses wxyz internally.
  ROS messages use xyzw. You must reorder when publishing.

- CustomDataset dummy camera:
  custom_dataset.py uses H36M-like extrinsics only for visualization; not true calibration.

- Depth translation depends on aligned depth:
  export_rgbd_from_bag.py produces paired frames by nearest timestamp.
  If your depth stream is not aligned to color (e.g., different optical centers), you need alignment upstream.

- Depth units:
  Most RealSense-like sources store depth in mm in a 16UC1 image.
  vis.py uses depth_scale (default often 0.001) to convert to meters.

- Translation-source placement in vis.py:
  Depth translation block sits inside the orientation-enabled path in current vis.py structure.
  For “depth translation only”, keep orientation estimation enabled (even if you don’t save it).
